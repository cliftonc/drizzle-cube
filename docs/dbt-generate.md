# `drizzle-cube dbt generate`

Generate Drizzle `pg-core` schema and Drizzle Cube files from local dbt
`manifest.json` and `catalog.json` artifacts.

This is a **local, artifact-first** command. It reads JSON files you already
have; it never clones a GitHub repo, never runs dbt, never connects to a
database, and never parses raw dbt project YAML or Jinja.

## Usage

```bash
npx drizzle-cube dbt generate \
  --manifest target/manifest.json \
  --catalog target/catalog.json \
  --dialect postgres \
  --out ./src/cubes/generated \
  --security-column organisation_id \
  --security-context organisationId
```

## Producing the artifacts

Run dbt outside this CLI to produce the two JSON files it consumes:

```bash
dbt docs generate    # produces target/manifest.json and target/catalog.json
# or
dbt compile          # produces target/manifest.json (no catalog)
```

`drizzle-cube dbt generate` accepts whatever paths you give it via
`--manifest` and `--catalog`.

## Options

| Option | Required | Description |
| --- | --- | --- |
| `--manifest <path>` | yes | Path to dbt `manifest.json`. |
| `--catalog <path>` | yes | Path to dbt `catalog.json` (carries column SQL types). |
| `--dialect <name>` | yes | Target SQL dialect. **v1 supports `postgres` only.** |
| `--out <dir>` | yes | Output directory for generated files. |
| `--security-column <name>` | no* | SQL column used for row-level security filtering. |
| `--security-context <name>` | no* | `securityContext` property bound to the column. |
| `--no-security` | no* | Explicitly opt out of cube-level security. |
| `--dry-run` | no | Report planned writes without writing anything. |
| `--check` | no | Fail (non-zero exit) when generated output would change or is stale. |
| `--force` | no | Overwrite non-generated files at expected paths. |
| `--config <path>` | no | Reserved for v1 (JSON config; not fully wired). |

\* Security is **required** but flexible: provide both `--security-column`
and `--security-context`, or `--no-security`, or answer the interactive
prompt (when stdin/stdout are TTYs). An empty prompt answer means
intentional no-security.

## Security modes

- **Filter** (`--security-column organisation_id --security-context organisationId`):
  every generated cube's `sql` filters `where: eq(table.organisationId,
  ctx.securityContext.organisationId as number)`. A model missing the
  configured column is **skipped with a warning** â€” a partially-unfiltered
  cube is never emitted when filter security is configured.
- **None** (`--no-security` or an empty interactive answer): cubes are
  emitted without a `where`, and a comment notes no cube-level security was
  requested. The CLI prints a warning to stderr.
- **Interactive**: when neither flag is provided and stdin/stdout are TTYs,
  you are prompted for the tenant column. A non-empty answer derives the
  context property (lowerCamel of the column) unless `--security-context` is
  also given.
- **Non-interactive without flags**: the command errors, requiring an
  explicit security choice. The generator never infers a security column.

Every generated file begins with `// Generated by drizzle-cube dbt generate.`
This ownership header lets the CLI safely overwrite its own files, refuse to
overwrite hand-written files (unless `--force`), and detect stale generated

## Primary keys and the baseline count measure

Primary keys are detected from `meta.drizzle_cube.primary_key: true` on a
column. The generator never infers a PK from a column merely being named
`id`.

- **Single-column PK**: emits a `primaryKey: true` dimension and a baseline
  `count` measure of `type: 'countDistinct'` on the PK column.
- **Composite PK**: emits **every** key column as a `primaryKey: true`
  dimension and a baseline `countDistinct` using a deterministic Postgres
  `concat_ws` expression, e.g.
  `sql\`count(distinct concat_ws('|', table.pkA, table.pkB))\``.
  A composite PK is **never** silently downgraded to plain `count`.

## Explicit measures

Additional measures are generated from `meta.drizzle_cube.measures` on a
model, e.g.:

```json
{
  "meta": {
    "drizzle_cube": {
      "measures": [
        { "name": "totalAmount", "type": "sum", "column": "amount", "format": "currency" }
      ]
    }
  }
}
```

Unsupported measure types, missing columns, or non-numeric aggregates are
**warned and skipped** (the measure is omitted, not the whole run).

## Relationships

Direct `belongsTo` joins are generated from dbt `relationships` tests. The
target cube is referenced by **string name** (`targetCube: 'Customers'`) to
avoid circular imports, while the `on` array uses direct Drizzle column
references (requiring an import of the target table). Relationships whose
source or target model/column was skipped are dropped with a warning.

## `--dry-run` and `--check`

- `--dry-run`: reports the planned creates/updates/deletes/conflicts without
  writing or deleting anything.
- `--check`: compares the **full expected file set** against the **full
  existing generated-header file set** under `--out`. It fails (non-zero
  exit) on changed, missing, conflicting, **or orphaned** files. A model
  deleted upstream leaves an orphaned generated file â€” `--check` reports
  that as drift. Use this in CI to catch drift.

In normal mode, stale generated-header files (no longer in the expected set)
are **deleted** with a warning.

## Warn-and-skip behaviour

For every input the generator does not fully support, the policy is
**warn-and-skip** (the column/join/measure/model is dropped with a visible
warning naming the model + column + offending value) or **warn-and-surface**
(an error/usage message) â€” never a silent default and never a `throw` on one
bad column. The one exception is **identifier collisions** (see below),
which throw because silent overwriting would lose generated output.

## Identifier collisions

If two models derive the same `tableExport`/`cubeName`/`cubeExport`/`fileName`
(e.g. `orders.total` and `orders_total` both normalize to `ordersTotal`), the
generator **throws** listing both offending models and the colliding
identifier, rather than silently overwriting one's output. Rename one model
before regenerating.

## Postgres type support

| Postgres type | Drizzle builder | Cube dimension type |
| --- | --- | --- |
| `smallint`, `integer`, `int`, `int4`, `serial`, `smallserial` | `integer` | `number` |
| `bigint`, `int8`, `bigserial` | `bigint({ mode: 'number' })` | `number` |
| `numeric`, `decimal` | `numeric` | `number` (runtime string) |
| `real`, `float4` | `real` | `number` |
| `double precision`, `float8` | `doublePrecision` | `number` |
| `text`, `varchar`, `character varying`, `char`, `character`, `uuid` | `text` | `string` |
| `boolean`, `bool` | `boolean` | `boolean` |
| `timestamp`, `timestamptz`, `time` | `timestamp` | `time` |
| `date` | `date` | `time` |
| `json`, `jsonb` | `jsonb` | `string` |

**Unsupported** (column is warned-and-skipped): arrays (`[]`), enums,
`bytea`, `interval`, `money`, geometry, network types (`inet`, `cidr`),
custom/user-defined types, and anything unrecognized. The generator never
substitutes a placeholder `text` for an unknown type.

## v1 limitations

- **Postgres only.** Other dialects are rejected before any file is read.
- **Local artifacts only.** No GitHub clone, no network, no running dbt.
- **No raw dbt project/YAML/Jinja parsing.** Provide `manifest.json` and
  `catalog.json`.
- **Materialized models only** (`table`, `view`, `incremental`). `ephemeral`
  and other materializations (`materialized_view`, `seed`, etc.) are skipped.
- **No sources, seeds, snapshots, exposures, metrics, or semantic models.**
- **Direct `belongsTo` joins only**, from dbt `relationships` tests. No
  reverse `hasMany`, many-to-many, or inferred join paths.
- **No merge-preserving manual edits.** Generated files start with the
  ownership header and are overwritten on regenerate; hand-written files at
  expected paths cause a conflict unless `--force`.
- **PK detection** relies on `meta.drizzle_cube.primary_key: true`.
- **`numeric`/`decimal`** store as strings at runtime but map to a `number`
  cube dimension.
