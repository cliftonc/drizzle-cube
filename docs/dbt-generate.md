# `drizzle-cube dbt generate`

Generate a Drizzle `pg-core` schema and Drizzle Cube cube files from local dbt
artifacts. The command reads `manifest.json` and `catalog.json` produced by
dbt, normalizes materialized Postgres models into a deterministic internal
representation, and emits:

- `schema.ts` — one `pgTable` per materialized model
- `cubes/<model>.ts` — one cube per model, using the public `defineCube` API
- `index.ts` — named re-exports + an `allCubes` array

It is **local and deterministic**: no network, no dbt runtime, no YAML/Jinja
parsing. Re-running on unchanged inputs produces a zero diff — which is what
makes `--check` meaningful.

## Usage

```bash
npx drizzle-cube dbt generate \
  --manifest target/manifest.json \
  --catalog target/catalog.json \
  --dialect postgres \
  --out ./src/cubes/generated \
  --security-column organisation_id \
  --security-context organisationId
```

## Required artifacts

Provide dbt's own artifacts (the CLI never runs dbt):

```bash
dbt docs generate    # produces target/manifest.json + target/catalog.json
# or
dbt compile          # produces target/manifest.json (catalog still needed)
```

## Options

| Option | Description |
| --- | --- |
| `--manifest <path>` | Path to dbt `manifest.json` (required) |
| `--catalog <path>` | Path to dbt `catalog.json` (required) |
| `--dialect <name>` | Target SQL dialect. v1 supports: `postgres` (required) |
| `--out <dir>` | Output directory (required) |
| `--security-column <name>` | Tenant/security column for row-level filtering |
| `--security-context <prop>` | Security context property compared against the column |
| `--no-security` | Explicitly skip cube-level security |
| `--dry-run` | Report planned changes without writing |
| `--check` | Detect drift (changed / missing / orphaned); exit non-zero on drift |
| `--force` | Overwrite non-generated files at expected paths |
| `--config <path>` | Reserved for a future config file |

## Security modes

Security is **explicit, never inferred**. Choose one of:

- **Filter** — `--security-column organisation_id --security-context organisationId`.
  Each cube's `sql` function gets a `where: eq(table.organisationId,
  ctx.securityContext.organisationId as <type>)`. A model missing the configured
  column is skipped with a warning — a partially unfiltered model is never
  emitted.
- **None** — `--no-security`, or an empty answer to the interactive prompt.
  Cubes are emitted without a `where` and a comment records the choice.
- **Interactive prompt** — when neither flag is given and stdin/stdout are TTYs,
  you are prompted for the tenant column. An empty answer is an intentional
  opt-out. In non-interactive (CI) invocations without flags, the command errors
  and asks for `--no-security` or both filter flags.

## Output layout

```
<out>/
├── schema.ts            # pgTable exports + `schema` + `Schema` type
├── index.ts             # named re-exports + `allCubes`
└── cubes/
    ├── orders.ts
    ├── customers.ts
    └── order-lines.ts
```

Every generated file starts with the ownership header
`// Generated by drizzle-cube dbt generate.` The writer refuses to overwrite a
file at an expected path that lacks this header unless `--force` is given.

## Warn-and-skip behavior

For every input v1 does not fully support, the generator **warns and skips** —
it never silently defaults or aborts the whole run on one bad column:

- Unsupported materializations (`ephemeral`, others) → `MODEL_SKIPPED`.
- Models with no catalog columns → `MODEL_SKIPPED`.
- Unsupported Postgres column types (arrays, enums, `bytea`, `interval`,
  `money`, geometry/network/custom/unknown) → `COLUMN_SKIPPED`. If the column
  was a primary key or the configured security column, the dependent output is
  dropped/skipped accordingly.
- Relationships pointing at a skipped model or a skipped column →
  `RELATIONSHIP_DROPPED`.
- Invalid explicit measure metadata → `MEASURE_SKIPPED`.

Identifier collisions (two models mapping to the same table export / cube name /
cube export / file name) **throw** — a stopped run is preferable to a silent
overwrite.

## `--dry-run`

Reports planned `created` / `updated` / `conflicts` and planned stale
deletions, without writing or deleting anything.

## `--check` (drift detection)

Compares the **full expected output set** against the **full existing
generated-header file set**, including removals:

- **changed** — expected files whose on-disk content differs.
- **missing** — expected files absent from disk.
- **orphaned** — on-disk generated files no longer expected (e.g. a model
  deleted upstream).

If any of these is non-empty, `--check` prints the drift and exits non-zero.
A `--check` that only diffed files it would currently write (and ignored files
it would no longer write) would pass on real drift — this command does not make
that mistake.

## Postgres type support

| Postgres type | Drizzle builder | Cube dimension type |
| --- | --- | --- |
| `smallint`, `integer`, `int`, `int4`, `serial`, `smallserial` | `integer` | `number` |
| `bigint`, `int8`, `bigserial` | `bigint({ mode: 'number' })` | `number` |
| `numeric`, `decimal` | `numeric` | `number` |
| `real`, `float4` | `real` | `number` |
| `double precision`, `float8` | `doublePrecision` | `number` |
| `text`, `varchar`, `character varying`, `char`, `character`, `uuid` | `text` | `string` |
| `boolean`, `bool` | `boolean` | `boolean` |
| `timestamp`, `timestamptz` | `timestamp` | `time` |
| `date` | `date` | `time` |
| `json`, `jsonb` | `jsonb` | `string` |

> `numeric`/`decimal` values are stored as strings by Drizzle; the cube
> dimension type is `number`. This is a documented v1 limitation.

## `meta.drizzle_cube` overrides

Primary keys and explicit measures are declared in dbt column/model metadata —
the generator never infers a PK from a column merely being named `id`.

- Column `meta.drizzle_cube.primary_key: true` marks a primary-key column.
  Composite primary keys are supported: every key column becomes a
  `primaryKey: true` dimension, and the baseline measure is `countDistinct`
  over a `concat_ws` expression of the key columns.
- Model `meta.drizzle_cube.measures` is an array of
  `{ name, type, column?, format?, description? }`. `type` must be one of
  `count`, `countDistinct`, `countDistinctApprox`, `sum`, `avg`, `min`, `max`,
  `runningTotal`, `number`. Numeric aggregates (`sum`/`avg`/`min`/`max`/
  `runningTotal`) require a referenced numeric column.

## v1 limitations

- **Local artifacts only.** No cloning of GitHub repos, no network.
- **Does not run dbt.** Provide `manifest.json` and `catalog.json` yourself.
- **Postgres only.** Other dialects are rejected before reading files.
- **No raw dbt project / YAML / Jinja parsing.**
- **No sources, seeds, snapshots, exposures, metrics, or semantic models.**
- **No `ephemeral` models** (warn-and-skip).
- **No arrays, enums, geometry, network, `bytea`, `interval`, `money`, or
  user-defined/custom column types** (warn-and-skip).
- **Only direct `belongsTo` joins** are derived from dbt `relationships` tests.
  Reverse `hasMany`, many-to-many, and inferred join paths are out of scope.
- **No merge-preserving manual edits.** Generated files are owned by the
  generator; edit the source dbt model and regenerate. Non-generated files at
  expected paths are treated as conflicts unless `--force` is given.
