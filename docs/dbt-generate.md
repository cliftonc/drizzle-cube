# dbt artifact generator

Generate Drizzle schema and Drizzle Cube definitions from local dbt artifacts:

```bash
npx drizzle-cube dbt generate \
  --manifest target/manifest.json \
  --catalog target/catalog.json \
  --dialect postgres \
  --out ./src/cubes/generated
```

The v1 generator is local-only and artifact-first. It does not run dbt, clone repositories, authenticate to GitHub, or parse dbt YAML/Jinja directly.

## Required artifacts

Provide a dbt `manifest.json` and `catalog.json`. You can produce them with commands such as:

```bash
dbt docs generate
```

or any dbt command in your workflow that writes `target/manifest.json` and `target/catalog.json`.

## Security modes

Drizzle Cube is security-first: production multi-tenant deployments should generate cube-level filters.

Interactive terminals prompt for security when no flags or config are supplied:

```text
Tenant/organisation filter column (blank for no cube-level security):
```

- Enter a column such as `organisation_id` to generate `where: eq(table.organisationId, ctx.securityContext.organisationId)`.
- Leave it blank only for public or single-tenant data.

CI and other non-interactive runs must be explicit:

```bash
npx drizzle-cube dbt generate \
  --manifest target/manifest.json \
  --catalog target/catalog.json \
  --dialect postgres \
  --out ./src/cubes/generated \
  --security-column organisation_id \
  --security-context organisationId
```

or opt out deliberately:

```bash
npx drizzle-cube dbt generate \
  --manifest target/manifest.json \
  --catalog target/catalog.json \
  --dialect postgres \
  --out ./src/cubes/generated \
  --no-security
```

`--no-security` prints a warning because generated cubes will not filter by `securityContext`.

## Output layout

The output directory contains owned generated files:

```text
src/cubes/generated/
  schema.ts
  cubes/
    customers.ts
    orders.ts
  index.ts
```

Every generated file starts with:

```ts
// Generated by drizzle-cube dbt generate.
```

Files with that header are overwritten deterministically on subsequent runs. Non-generated files are not overwritten unless `--force` is supplied. Stale generated files are not deleted in v1.

## Configuration

Optional JSON config can be loaded with `--config`:

```json
{
  "security": { "mode": "column", "column": "organisation_id", "context": "organisationId" },
  "models": {
    "orders": {
      "cubeName": "Orders",
      "tableExportName": "orders",
      "columns": {
        "customer_id": { "propertyName": "customerId", "dimensionName": "customerId", "dimensionType": "number" }
      },
      "measures": [
        { "name": "totalAmount", "type": "sum", "column": "amount", "title": "Total Amount" }
      ]
    }
  },
  "typeOverrides": {
    "money": { "drizzleBuilder": "numeric", "dimensionType": "number" }
  }
}
```

Precedence is: CLI security flags > config security > dbt `meta.drizzle_cube` > naming/type defaults. For non-security settings, model and column config can override dbt metadata, and explicit config measures are appended to dbt metadata measures.

## Supported v1 scope

- Dialect: Postgres only.
- dbt nodes: materialized models only (`table`, `view`, `incremental`, `materialized_view`). Sources, seeds, snapshots, and ephemeral models are out of scope.
- Types: common Postgres scalar types (`integer`, `bigint`, `numeric`, `real`, `double precision`, text/varchar/uuid, boolean, timestamp/date/time, json/jsonb). Unsupported types fail with a clear error unless configured through `typeOverrides`.
- Measures: baseline `count`/`countDistinct`; additional measures only from `meta.drizzle_cube.measures` or generator config. Numeric columns are not inferred as sums or averages.
- Joins: direct `belongsTo` joins from dbt `relationships` tests where both models and concrete columns can be resolved.

## Check and dry-run

Use `--dry-run` to print what would be created or updated without writing files:

```bash
npx drizzle-cube dbt generate ... --dry-run
```

Use `--check` in CI to fail when committed generated output is stale:

```bash
npx drizzle-cube dbt generate ... --check
```

Use `--force` only when replacing existing non-generated files intentionally.

## Out of scope for v1

- GitHub sync, clone, auth, or PR automation.
- Running dbt commands.
- YAML/Jinja parsing.
- Non-Postgres adapters.
- Reverse join, many-to-many, or fan-out inference.
